## What this document is (plain English)

This is **how code is written** in this project — naming rules, formatting, patterns. Boring but essential.

**Why it matters:** Without conventions, every file looks different, reviews get messy, and you end up with three ways to do the same thing. With conventions, every new file follows a template, Claude Code's output is predictable, and the codebase feels like it was written by one person — even though it was written over 8 weeks with AI help.

**When Claude Code uses this document:** every time it writes code. Names, imports, error handling, file structure — all follow these rules.

# Coding Conventions

## Python (backend)

### Style
- Python 3.12+
- Formatter: `ruff format` (not black, not autopep8)
- Linter: `ruff check`
- Type checker: `mypy --strict` on `app/` (not tests)
- Line length: 100
- Double quotes for strings, single only to avoid escapes

### Naming
- `snake_case` for functions, variables, modules
- `PascalCase` for classes, Pydantic models, SQLAlchemy models, Enums
- `SCREAMING_SNAKE_CASE` for module-level constants
- Private with single leading underscore: `_internal_helper`
- Never use `l`, `O`, `I` as single-letter names

### Typing (non-negotiable)
- Every function signature fully typed, including return type
- `-> None` explicit
- Use `from __future__ import annotations` at top of every `app/` file
- Prefer `list[str]` over `List[str]` (3.12 syntax)
- Use `| None` over `Optional[X]`
- Use Pydantic models over raw dicts for anything crossing a boundary

### Imports
Order: stdlib → third-party → local. Blank line between groups. `ruff` handles this.

```python
from __future__ import annotations

import asyncio
from datetime import datetime, timezone
from uuid import UUID

from fastapi import APIRouter, Depends, HTTPException
from sqlalchemy.ext.asyncio import AsyncSession

from app.core.database import get_db
from app.core.security import get_current_user
from app.models.user import User
from app.schemas.booking import BookingCreate, BookingOut
from app.services.booking_service import create_booking
```

Never `from app import *`. Never relative imports (`from ..models import`).

### Async rules
- Every endpoint handler is `async def`
- Every DB call is awaited (`await session.execute(...)`, `await session.commit()`)
- Never mix sync and async in the same function
- Background jobs via `asyncio.create_task` only inside request handlers; for cron-like jobs use APScheduler

### Error handling
- Never `except:` bare. Always `except SpecificException:`.
- Use custom exceptions from `app/core/exceptions.py`: `NotFoundError`, `ValidationError`, `AuthorizationError`, `BusinessLogicError`, etc.
- FastAPI exception handlers map custom exceptions to HTTP responses
- Routers never raise `HTTPException` directly — raise custom exceptions, let handlers translate
- Services never raise `HTTPException`. Services are HTTP-agnostic.

### SQLAlchemy patterns
- Always use `mapped_column()` not `Column()` (2.0 style)
- Use `Mapped[T]` annotations on every column
- Relationships typed with `Mapped[list["Other"]]`
- Reserved word `metadata` → use `mapped_column("metadata", ..., name="metadata")` with Python attr `meta` or `extra_metadata`
- Always call `await session.flush()` before reading an autogenerated ID
- Always `await session.commit()` explicitly — no implicit commits
- Wrap risky blocks in try/except with `await session.rollback()` in except

### Pydantic patterns
- v2 syntax only. `model_config = ConfigDict(...)`.
- `Field(..., description="...")` for every non-obvious field
- Use `model_validator` for cross-field validation
- Separate models for input/output: `BookingCreate` (input), `BookingUpdate` (input), `BookingOut` (output), `BookingInternal` (internal only). Never reuse.
- `from_attributes=True` on Out models that wrap ORM objects

### Logging
- `import logging` + `logger = logging.getLogger(__name__)` at top of every module
- Levels: `debug` for deep trace, `info` for normal flow, `warning` for expected-but-unusual, `error` for unexpected, `critical` for alert-worthy
- Never log PII (emails, phone numbers) — use IDs

### Docstrings
- Module-level docstring describing what the module is for
- Public functions: one-line summary. Multi-line only for non-obvious behavior.
- No type restating — types are in the signature.

### Testing
- `pytest` + `pytest-asyncio`
- Test file path mirrors module path: `app/services/booking_service.py` → `tests/unit/test_booking_service.py`
- Fixtures in `conftest.py`
- Each test has one reason to fail
- Name: `test_<action>_<expected_outcome>_<condition>`, e.g., `test_create_booking_raises_when_outside_hours`

## TypeScript (frontend)

### Style
- TypeScript strict mode on (`"strict": true`)
- Formatter: Prettier (or Biome, pick one)
- Linter: ESLint with Next.js config (or Biome)
- Line length: 100
- Double quotes, semicolons on

### Naming
- `camelCase` for variables, functions, hooks
- `PascalCase` for components, types, interfaces, enums
- `UPPER_SNAKE_CASE` for module-level constants
- Component files: `kebab-case.tsx` containing `PascalCase` component
- Hook files: `use-<thing>.ts` containing `useThing` hook

### Typing (non-negotiable)
- Never `any`. Use `unknown` if truly unknown, then narrow.
- Never `as` casting unless unavoidable — refactor to make types flow naturally
- Prefer `type` aliases for unions/primitives, `interface` for object shapes that may extend
- Every component's props typed in an adjacent `type Props = { ... }`
- Zod schemas for anything crossing the API boundary — types derived via `z.infer<typeof schema>`

### Imports
Order: `react`/`next` → third-party → `@/*` aliases → relative. Prettier/Biome enforces.

```tsx
"use client";

import { useState } from "react";
import Link from "next/link";

import { useForm } from "react-hook-form";
import { zodResolver } from "@hookform/resolvers/zod";

import { Button } from "@/components/ui/button";
import { useBookings } from "@/lib/hooks/use-bookings";
import { bookingSchema } from "@/lib/schemas/booking";

import type { Booking } from "@/types/models";
```

### Component rules
- Server component by default; add `"use client"` only when needed (state, effects, browser APIs, event handlers)
- One component per file, exported as **default** for pages/layouts, **named** for everything else
- Props destructured in signature: `function MyThing({ foo, bar }: Props)`
- No inline arrow components inside other components (rebuilds every render)
- Never fetch inside components — use TanStack Query hooks

### Data fetching
- All API calls go through `lib/api/*`
- All components read via TanStack Query hooks in `lib/hooks/*`
- Mutations invalidate queries, not refetch manually
- Loading states: skeletons, not spinners, where possible
- Errors handled at component level with toast + fallback UI

### Forms
- React Hook Form + Zod, always
- Zod schema in `lib/schemas/`, imported by form + API call
- Server validation errors mapped back to form fields via `setError`

### State management
- **Server state** → TanStack Query
- **Client state shared across components** → Zustand
- **Local component state** → `useState`
- Never use React Context for anything that changes (causes re-renders); only for stable values (theme, auth user reference)

### Styling
- Tailwind only. No CSS modules, no styled-components, no inline `style={{}}` unless dynamic computed value
- Use `cn()` utility for conditional classes (from `lib/utils/cn.ts`)
- shadcn components get variants — don't re-style, extend via `className`
- Dark mode: use Tailwind `dark:` prefix, theme toggle via `next-themes`

### File header (for every component file)
```tsx
"use client"; // only if needed

// imports

type Props = { ... };

export function MyComponent({ ... }: Props) {
  // hooks first
  // derived values next
  // handlers
  // render
}
```

## Git

### Commits
- Conventional commits format: `type(scope): message`
- Types: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `style`, `perf`
- Scope optional but preferred: `feat(auth): add refresh token endpoint`
- Present tense, imperative mood: "add" not "added"
- Body explains *why*, not *what* (diff shows what)

### Branches
- `main` is deployable at all times
- Feature branches: `feat/short-description`
- Fix branches: `fix/short-description`
- Never commit directly to `main` on real teams; solo it's fine but squash-merge from branches is better habit

### PRs (for the habit, even solo)
- Opens one logical change
- Description: what + why + how to test
- Self-review before "merging"

## Commenting

- Code should be self-explanatory; comments explain *why*, not *what*
- TODO comments: `# TODO(emon): reason`, never bare `# TODO`
- Never leave commented-out code. Git has history.
- Surprising behavior → comment required

---

## Secrets + safety

- Never log full request bodies (may contain PII, passwords)
- Never commit `.env`
- Never print secrets, even in debug
- Mask tokens/keys in admin UIs (`sk_live_****xyz`)
- Run `ruff check --select S` (bandit rules) before merging

---

## Performance defaults

- Index every foreign key column
- Index every column in a frequent `WHERE` clause
- `SELECT` only columns you need
- Paginate any list endpoint (default limit 20, max 100)
- Never `N+1` query — use `selectinload()` or `joinedload()`
- For chat/voice endpoints: target p95 < 2s

---

## Accessibility (frontend)

- Every interactive element has a role or uses a native element
- Forms: `<label>` for every `<input>`, even if visually hidden
- Buttons have text or `aria-label`
- Color is not the only indicator (use icons + text too)
- Keyboard: Tab reaches everything, Enter/Space activates buttons

## What Claude Code must never do

- Hardcode secrets
- `except: pass`
- Commit to `main` without my review
- Skip types to "get it working"
- Use jQuery (shouldn't come up, but just in case)
- Use class components in React
- Use `any` type
- Install a new library without adding it to `pyproject.toml` / `package.json` and telling me why
- Run database migrations without showing me the SQL first
- Make a Stripe or Vapi API call in tests (mock them)
